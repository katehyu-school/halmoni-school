# HQ Kids 모바일 앱 방향 제안 (2026-06-15)

> 참조: `hq-mobile-prototype.html`, `한국어 교육 모바일 앱 UI_UX 전략 설계.pdf`, `docs/MOBILE_CONTENT_STRATEGY.md`, `data/elem/level1/unit0N.json`
> 핵심 원칙: [[fix-over-rebuild-preference]] — 새 콘텐츠 제작이 아니라 **이미 완성된 Level 1 JSON을 모바일 게임 엔진으로 재노출**

---

## 1. 현재 상태 진단

### 프로토타입 (Kids 트랙)
- 모험 맵 5노드 중 **"블록 조합" 1개만 동작** — `GAME` 배열 3라운드가 HTML에 하드코딩
- 자음(빨강)+모음(초록) 2슬롯 탭 방식은 이미 구현되어 있고, CSS 토큰(`--block-con/vow/bat`)도 준비됨
- 나머지 노드(단어 만들기, 받침 쌓기)와 하단 네비(게임/내 별)는 토스트만

### HQ(틴+성인) 트랙과의 격차
- HQ 트랙은 `data/nhs/L1/epNN.json`을 fetch해서 Watch→Learn→Practice→Review 4단계를 자동 렌더링
- Kids 트랙은 그런 데이터 연동이 전혀 없음 — 콘텐츠가 1개 미니게임뿐

### 그런데 — Level 1 JSON은 이미 "모바일 게임 데이터"로 설계되어 있음
`korean-app_v2.html`의 `data/elem/level1/unit00~09.json`을 열어보면, 프로토타입의 `GAME` 배열과 거의 동일한 구조의 게임 데이터가 유닛마다 이미 들어 있음:

| 유닛 | 게임 데이터 | 구조 |
|------|------------|------|
| 00 (가나다라) | `recognition_game` | 탭 매칭 |
| 01 (기본 모음) | `sound_tap_game`, `classify_game` | 듣고 탭 / 2분류 |
| 02 (복합 모음) | `groups` 내 라운드 | 탭 매칭 |
| 03 (기본 자음) | `first_sound_game` | 듣고 탭 |
| 04 (쌍자음/거센소리) | `compare_game` | 2분류 |
| 06 (받침) | `block_game` (받침 쌓기) | **3슬롯 블록 빌드** — 프로토타입 GAME의 2슬롯 확장판, 데이터 이미 존재 |
| 07 (안녕하세요?) | `word_card_game` | 카드 매칭 |
| 09 (이게 뭐예요?) | `picture_game` | 그림-단어 매칭 |

→ 즉 **9개 유닛 분량의 모바일 게임 콘텐츠가 이미 작성되어 있고, 프로토타입에 연결만 하면 됨.**

## 2. 방향 제안: "야옹이 모험 맵 = Level 1 진행도 그대로"

### 큰 그림
현재 5개의 의미 없는 placeholder 노드를 버리고, **Level 1 unit00~09의 실제 커리큘럼 순서를 모험 맵으로 그대로 시각화**한다. 웹에서 배운 순서(모음→자음→복합모음→받침→문장)가 곧 모바일 퀘스트 순서가 되므로, 학생은 "웹에서 배운 걸 모바일로 복습"하는 자연스러운 흐름을 갖게 된다 (HQ 트랙의 "웹 연동 복습" 철학과 동일선상).

```
🔤모음1(u01) → 🔤모음2(u02) → 🔠자음1(u03) → 🔠자음2(u04)
  → 🧱블록조합(u00 음절) → 🏗️받침쌓기(u06, 3층)
  → 🧩단어만들기(u07/09 문장 카드)
```

### 게임 엔진을 4종 템플릿으로 일반화
유닛마다 새 코드를 짜지 않고, 기존 `gTap`/슬롯 로직을 4개 재사용 템플릿으로 확장한다:

1. **tap-match** (`sound_tap_game`, `first_sound_game`, `recognition_game`) — 프로토타입 `gListen()`+선택 탭 로직 거의 그대로 재사용
2. **classify** (`classify_game`, `compare_game`) — 2개 카테고리 버튼 + 항목 탭 분류 (신규, 단순)
3. **block-build** (`block_game` 받침쌓기, 기존 음절조합) — 기존 2슬롯(자음+모음) 로직을 **3슬롯(자음/모음/받침)**으로 확장. CLAUDE.md에 이미 설계된 "3층 블록(상/중/하)" 그대로 구현
4. **card-match** (`word_card_game`, `picture_game`) — 이미지/단어 카드 탭 매칭

각 유닛 JSON의 `*_game` 필드를 fetch → 타입에 맞는 템플릿으로 렌더링. HQ 트랙의 `normalizeEp()`와 같은 패턴으로 `normalizeUnitGame(n)` 함수 하나만 추가하면 됨.

### 비주얼/마스코트
- PDF가 제안하는 "3D 픽사 캐릭터가 퀘스트 안내"는 **야옹이(🐱)** 캐릭터로 이미 확정되어 있음 (Level1 u03 첫 등장 → Level2 u01 재등장 캐릭터 연결 설계 기존 완료)
- 지금은 이모지 야옹이로 시작 → 추후 PDF의 "3차 고도화(크로스 미디어, 95%)" 단계에서 픽사풍 일러스트로 교체. 지금 단계에서 이미지 제작에 투자하지 않는다.
- 블록 색상(자음=빨강/모음=초록/받침=파랑)은 프로토타입 CSS에 이미 토큰화되어 있어 추가 디자인 작업 불필요.

### 진행도 ("내 별" 탭)
- 라운드 완료 시 ⭐ +1은 이미 동작 (`kStars`) — 이를 `localStorage`에 프로필별로 저장(korean-app_v2의 My Space `ms_{name}_*` 네임스페이스 패턴 재사용)
- "내 별" 탭 = 유닛별 별 개수 + 모험 맵 잠금 해제 상태 표시. 별도 백엔드 없이 PWA 로컬 저장으로 충분 (1단계 PWA 전략과 일치)

## 3. 단계별 로드맵

| 단계 | 내용 | 비고 |
|------|------|------|
| A | `normalizeUnitGame()` + tap-match/classify 템플릿 구현 → unit00~04 모험 맵 노드 5개 연결 | 신규 콘텐츠 0, 기존 코드 확장만 |
| B | block-build를 3슬롯(자음/모음/받침)으로 확장 → unit06 받침쌓기 연결 | CLAUDE.md 기존 설계 그대로 구현 |
| C | card-match 템플릿 → unit07/09 문장 카드 연결 | Level 2 진입 전 다리 역할 |
| D | "내 별" 탭 실제 연동 + 모험 맵 잠금 로직 + PWA 매니페스트 | 기존 로드맵 항목과 병합 |
